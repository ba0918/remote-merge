# CLI Hunk Merge 実装計画

**Created:** 2026-04-06  
**Status:** 🟢 Complete  

---

## 概要

CLI merge コマンドに `--hunks` オプションを追加し、hunk 単位の部分マージを可能にする。
LLM エージェントが `diff` の JSON 出力から hunk インデックスを取得し、特定の hunk のみをマージするユースケースに対応する。

## 背景

- TUI 側には既に hunk merge 機能がある（`app/hunk_ops.rs` + `diff/engine.rs::apply_hunk_to_text()`）
- CLI の merge は現在ファイル単位の丸ごとマージのみ
- LLM エージェントワークフロー: `diff --format json` → hunk 確認 → `merge --hunks 0,2` で選択的マージ

## 設計

### CLI インターフェース

```
remote-merge merge --left local --right develop src/foo.rs --hunks 0,2,5
```

- `--hunks`: カンマ区切りの hunk インデックス（0-based、diff JSON の `hunks[].index` に対応）
- `--hunks` 指定時は **paths が1つのみ** であること（複数パスはエラー）
- `--hunks` なしの既存動作は一切変更しない

### マージ方向

既存の merge と同じ: `--left` → `--right` の方向（left の内容を right に書き込む）。
hunk merge もこの方向に従う。つまり `--hunks 0,2` は「hunk 0, 2 の変更を right 側に反映する」。

### 内部フロー

1. `MergeArgs` に `hunks: Option<Vec<usize>>` フィールド追加
2. バリデーション: `--hunks` 指定時に paths が1つでなければエラー
3. hunk merge 専用パス:
   a. 両側のファイル内容を取得
   b. `diff/engine.rs::compute_diff()` で diff 計算 → `merge_hunks` を取得
   c. 指定された hunk インデックスを検証（範囲外ならエラー）
   d. `diff/engine.rs::apply_hunk_to_text()` を hunk ごとに順次適用
   e. 結果を `write_file_bytes()` で書き込み
4. JSON/text 両方の出力に対応

### hunk 適用の順序問題

複数 hunk を適用する場合、先の hunk 適用で行番号がずれる。
**解決策**: hunk を **後ろから前へ**（インデックス降順）適用する。
後ろの hunk を先に適用すれば、前の hunk の行番号に影響しない。

### 出力

既存の `MergeFileResult` を拡張し、hunk merge の情報を含める:

```json
{
  "path": "src/foo.rs",
  "status": "merged",
  "direction": "left_to_right",
  "hunks_applied": [0, 2, 5],
  "hunks_total": 8
}
```

## 実装ステップ

### Step 1: ドメイン層 — hunk 選択マージ関数 (pure function)
**ファイル**: `src/diff/engine.rs`

`apply_hunk_to_text()` は1 hunk ずつ適用する既存関数。
新たに `apply_selected_hunks()` を追加:
- 入力: 元テキスト(`&str`)、全行リスト(`&[DiffLine]`)、merge_hunks 一覧(`&[DiffHunk]`)、適用する hunk インデックス(`&[usize]`)、方向(`HunkDirection`)
- 処理: インデックス降順ソート → 各 hunk について `DiffHunk::lines()` で hunk 行を取得し `apply_hunk_to_text()` を順次適用
- 出力: `Result<String>` — マージ後テキスト（範囲外インデックスは `Err`）
- **純粋関数** — テスト容易

### Step 2: サービス層 — hunk merge オーケストレーション
**ファイル**: `src/service/merge_flow.rs`（hunk merge パイプライン部分を新規追加）

`execute_hunk_merge()` を追加（CLI 固有のロジックは含めない。引数バリデーション済みの値を受け取る）:
- **sensitive ファイルチェック**: `is_sensitive()` で `.env`, `*.pem` 等のファイルを検出し、`--force` なしなら警告付きエラーを返す（既存の全体マージと同じガード）
- **symlink/バイナリ ガード**: 対象ファイルが symlink またはバイナリの場合はエラーを返す（hunk merge はテキストファイル専用）
- 両側のファイル内容取得
- diff 計算 → `DiffResult::Equal` ならスキップ（"no changes" として成功を返す）
- merge_hunks 取得
- hunk インデックス検証（範囲外ならエラー）
- `apply_selected_hunks()` 呼び出し
- 楽観的ロック検証（mtime 再チェック）
- バックアップ（既存の仕組みを流用）
- ファイル書き込み

### Step 3: CLI 層 — 引数パース + バリデーション + 分岐
**ファイル**: `src/cli/merge.rs`, `src/main.rs`

- `MergeArgs` に `hunks: Option<Vec<usize>>` 追加
- clap の引数定義に `--hunks` 追加（value_delimiter=','）
- `validate_merge_args()` に追加バリデーション:
  - `--hunks` + 複数パス → エラー
  - `--hunks` + `--delete` → エラー（意味的に矛盾）
- `run_merge()` 内で `args.hunks.is_some()` を判定し hunk merge 専用パスに分岐
- CLI 層は引数パース・バリデーション・出力フォーマット選択のみを担当。ビジネスロジックは Step 2 の `execute_hunk_merge()` に委譲する

### Step 4: 出力フォーマット
**ファイル**: `src/service/output.rs`, `src/service/types.rs`

- `MergeFileResult` に以下のフィールドを追加:
  - `hunks_applied: Option<Vec<usize>>` — 適用された hunk インデックス
  - `hunks_total: Option<usize>` — 全 hunk 数
  - `direction: Option<String>` — マージ方向（`"left_to_right"` / `"right_to_left"`）
  - 全て `#[serde(skip_serializing_if = "Option::is_none")]` で既存出力を壊さない
- JSON/text 出力でこれらのフィールドを表示
- `--hunks` なしの通常マージでは全て `None`（後方互換性維持）

### Step 5: テスト
**ファイル**: 各ステップのモジュール内 `#[cfg(test)] mod tests`

- Step 1: `apply_selected_hunks()` の単体テスト
  - 複数 hunk の降順適用で行番号が正しく処理される
  - 範囲外インデックスでエラー
  - 空リストで元テキストがそのまま返る
  - 全 hunk 指定で全体マージと同等の結果になる
- Step 2: `execute_hunk_merge()` のテスト
  - バイナリファイル指定時にエラー
  - symlink ファイル指定時にエラー
  - `DiffResult::Equal`（変更なし）でスキップ
  - sensitive ファイル + `--force` なしでエラー
- Step 3: CLI 引数バリデーションテスト
  - `--hunks` + 複数パス → エラー
  - `--hunks` + `--delete` → エラー
  - `--hunks 0,2,5` のパース正常系
- Step 4: 出力フォーマットテスト
  - hunk merge 結果の JSON シリアライズ（`hunks_applied`, `hunks_total`, `direction` 含む）
  - 通常マージ結果で追加フィールドが `None`（出力に含まれない）ことの確認

### Step 6: ドキュメント・Skill 更新
- `--hunks` オプションの CLI ヘルプ追加
- Skill ファイルに hunk merge の使用例追加

## スコープ補足

- `--dry-run` + `--hunks` の組み合わせ: 既存の dry-run 分岐に自然に乗る設計のため追加コストが小さく、スコープ内で対応する（Step 2 の書き込みスキップで実現）

## 非スコープ

- 複数ファイル一括 hunk merge（将来の拡張）
- TUI 側の変更（既存の hunk merge は変更しない）

---

## リスク・考慮事項

- **hunk インデックスの安定性**: diff の hunk インデックスは diff 実行ごとに同じファイル内容なら安定。ただしファイルが変更されると hunk 構成が変わる → 楽観的ロック（mtime チェック）で検知
- **`--dry-run` との組み合わせ**: 自然に対応可能（書き込みスキップ + 結果出力のみ）
